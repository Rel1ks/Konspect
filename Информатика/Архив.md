## Введение

> [!abstract] Определение
> **Архив** — это файл (или группа файлов), который хранит один или несколько сжатых файлов вместе с метаданными: именами, размерами, датами создания, правами доступа и контрольными суммами. Архивация позволяет объединять множество файлов в единый контейнер и уменьшать их суммарный объём.

Архив — это уникальная структура данных, позволяющая объединять несколько файлов воедино. Основная суть архивации — ==оптимизация хранения==, то есть упорядочивание и **сжатие исходных данных без потерь**. Это означает, что после распаковки файл будет абсолютно идентичен оригиналу, — ни один бит информации не будет утерян.

> [!quote] Ключевая идея
> Сжатие без потерь (lossless compression) — это класс методов кодирования, при котором исходные данные могут быть **полностью восстановлены** из сжатого представления. В отличие от сжатия с потерями (например, JPEG или MP3), здесь не происходит отбрасывания «неважной» информации.

Процесс сжатия основан на поиске и устранении избыточности в данных. Любые данные, как правило, содержат повторяющиеся фрагменты, предсказуемые закономерности или неравномерное распределение символов. Именно эти особенности алгоритмы сжатия используют для сокращения объёма информации.

---

## Зачем нужна архивация?

> [!info] Основные причины
> Архивация решает сразу несколько задач, которые важны как для повседневного использования, так и для профессиональной работы с данными.

| Причина | Описание | Пример |
|:---|:---|:---|
| **Экономия места** | Сжатие уменьшает объём данных, что позволяет хранить больше информации на носителе | Текстовый файл 10 МБ сжимается до ~2 МБ |
| **Ускорение передачи** | Меньший объём — быстрее загрузка и скачивание по сети | Скачивание архива в 3 раза быстрее оригинала |
| **Объединение файлов** | Несколько файлов и папок упаковываются в один контейнер | Отправка папки с проектом одним файлом |
| **Защита данных** | Архивы могут быть зашифрованы паролем | ZIP с паролем для конфиденциальных документов |
| **Сохранение метаданных** | Сохраняются структура каталогов, права доступа, даты | Бекап папки с сохранением всей иерархии |

## Общая классификация методов сжатия

> [!tip] Два больших класса
> Все методы сжатия без потерь делятся на две основные категории: **словарные** (ищут повторяющиеся последовательности) и **энтропийные** (используют частоту символов). Самые эффективные алгоритмы комбинируют оба подхода.

```
Методы сжатия без потерь
├── Словарные методы
│   ├── LZ77 (1977)
│   ├── LZ78 (1978)
│   ├── LZW (1984)
│   └── LZMA (2001)
├── Энтропийное кодирование
│   ├── Кодирование Шеннона-Фано (1949)
│   ├── Кодирование Хаффмана (1952)
│   └── Арифметическое кодирование (1979)
└── Гибридные методы
    ├── Deflate (LZ77 + Хаффман)
    ├── gzip (Deflate + заголовки)
    ├── PNG (фильтрация + Deflate)
    └── Zstandard / Brotli (современные)
```

> [!note] Формула Шеннона
> Американский математик Клод Шеннон в 1948 году доказал, что **минимальная средняя длина кода** для символа не может быть меньше его энтропии:
>
> $$H = -\sum_{i=1}^{n} p_i \cdot \log_2(p_i)$$
>
> где $p_i$ — вероятность появления $i$-го символа. Эта формула задает ==теоретический предел== сжатия для любого алгоритма.

## ZIP-архив

> [!info] Историческая справка
> **ZIP-архив** придумал американский программист **Фил Кац** (Phil Katz) в **1989 году**. Он создал формат `.ZIP` и программу **PKZIP** через свою компанию PKWARE как быструю, открытую альтернативу закрытому формату ARC.
>
> Формат ZIP стал стандартом де-факто для сжатия файлов и поддерживается практически всеми операционными системами мира.

### Как устроен ZIP-файл

Структура ZIP-архива состоит из трёх основных секций, расположенных последовательно:

```
┌─────────────────────────────────────┐
│  Локальные заголовки файлов          │
│  (для каждого сжатого файла)        │
│  ┌─────────────────────────────┐    │
│  │ Заголовок + сжатые данные    │    │
│  │ Заголовок + сжатые данные    │    │
│  │ ...                         │    │
│  └─────────────────────────────┘    │
├─────────────────────────────────────┤
│  Центральный каталог                 │
│  (указатели на все файлы)           │
│  (используется для быстрого доступа) │
├─────────────────────────────────────┤
│  Конец центрального каталога (EOCD)  │
│  (метаданные архива в целом)        │
└─────────────────────────────────────┘
```

### Ключевые особенности ZIP

- **Множественные алгоритмы**: ZIP поддерживает несколько методов сжатия, но ==Deflate== является стандартным
- **Многотомные архивы**: большой архив можно разбить на несколько частей (например, `archive.z01`, `archive.z02`, `archive.zip`)
- **Шифрование**: поддержка AES-256 для защиты содержимого
- **Устойчивость к повреждениям**: благодаря центрального каталога, расположенного в конце файла, можно восстановить отдельные файлы даже при повреждении архива

> [!warning] Важно
> ZIP-файл **не является** средством долгосрочного хранения. Для бекапов, которые должны храниться годами, лучше использовать специальные форматы или несколько независимых копий.

## Алгоритм Лемпеля-Зива (LZ77)

> [!abstract] Краткая справка
> **LZ77** — словарный алгоритм сжатия, разработан **Авраамом Лемпелем** и **Яаковом Зивом** в **1977 году**. Относится к классу словарных методов сжатия и стал одним из самых влиятельных алгоритмов в истории.

### Принцип работы

> [!tip] Суть алгоритма
> Алгоритм просматривает данные через «скользящее окно» и ищет повторяющиеся фрагменты. Если находит повтор — он не записывает его заново, а указывает: **где** этот фрагмент уже был (смещение) и **какой** он длины (длина).

Алгоритм LZ77 использует три типа элементов (токенов) для кодирования данных:

| Тип токена | Значение | Описание |
|:---|:---|:---|
| **Буквальный символ** | `(0, 0, C)` | Символ `C` записывается как есть (повтор не найден) |
| **Ссылка на совпадение** | `(D, L, C)` | Повтор фрагмента длины `L` на расстоянии `D` назад, затем символ `C` |

> [!note] Скользящее окно
> LZ77 работает с двумя буферами:
> - **Буфер поиска** (Search Buffer) — уже обработанные данные, в которых ищутся совпадения
> - **Буфер упреждения** (Lookahead Buffer) — ещё не обработанные данные, которые нужно сжать
>
> Типичный размер буфера поиска — **32 КБ** (32768 байт).

### Подробный пример

> [!example] Пошаговое сжатие строки
> Рассмотрим сжатие строки `АБАБАБАБ`:
>
> **Шаг 1.** Буфер поиска пуст. Берём символ `А` — записываем литерал: `(0, 0, А)`
>
> **Шаг 2.** В буфере поиска: `А`. Следующий символ `Б` не совпадает с `А` — записываем литерал: `(0, 0, Б)`
>
> **Шаг 3.** В буфере поиска: `АБ`. Ищем `АБАБАБ` в буфере. Находим `АБ` на расстоянии 2, длина совпадения = 6:
> `→ (2, 6, )` — «повтори 6 символов, начиная с позиции на 2 назад»
>
> **Результат:**
> ```
> Было:    А Б А Б А Б А Б   (8 символов)
> Стало:   (0,0,А) (0,0,Б) (2,6)   (3 токена)
> ```

### Ещё один пример

> [!example] Сжатие осмысленного текста
> ```
> Исходная строка:
> "СЛОН СЛОН СЛОВАРЬ"
>
> После сжатия LZ77:
> (0,0,С)(0,0,Л)(0,0,О)(0,0,Н)(0,0, )(5,5)(4,2)АРЬ
>
> Объяснение:
> 1. С, Л, О, Н, пробел → записываем как литералы
> 2. "СЛОН " повторяется → (5,5) = "назад на 5, длина 5"
> 3. "СЛ" из "СЛОВАРЬ" совпадает с началом "СЛОН" → (4,2) = "назад на 4, длина 2"
> 4. "ОВАРЬ" → дозаписываем как литералы
> ```

### Преимущества и ограничения

- [x] Хорошо сжимает текст с повторяющимися фрагментами
- [x] Адаптивный — не требует предварительного анализа данных
- [x] Прост в реализации
- [ ] Плохо работает на случайных данных (энтропийных)
- [ ] Эффективность ограничена размером окна

## Алгоритм Хаффмана

> [!abstract] Краткая справка
> **Кодирование Хаффмана** — метод энтропийного кодирования, разработан **Дэвидом Хаффманом** в **1952 году**. Алгоритм строит ==префиксное кодовое дерево==, назначая короткие коды частым символам и длинные — редким.

### Принцип работы

> [!tip] Три этапа алгоритма
> 1. **Подсчёт частот**: посчитать, сколько раз встречается каждый символ
> 2. **Построение дерева**: объединять два самых редких узла в новый, пока не останется одно дерево
> 3. **Кодирование**: пройти от корня к каждому символу, записывая направление (лево = `0`, право = `1`)

> [!note] Префиксный код
> Код Хаффмана обладает важным свойством: ==ни один код не является префиксом другого==. Это позволяет однозначно декодировать поток битов без разделителей между кодами.

### Подробный пример

> [!example] Построение дерева Хаффмана
> Строка: `ААААБВ`
>
> **Этап 1. Подсчёт частот:**
>
> | Символ | Частота |
> |:---:|:---:|
> | А | 4 |
> | Б | 1 |
> | В | 1 |
>
> **Этап 2. Построение дерева:**
> ```
> Шаг 1:  А(4)   Б(1)   В(1)
>                   ↓ объединяем два самых редких
> Шаг 2:  А(4)        БВ(2)
>                      ↓ объединяем А(4) и БВ(2)
> Шаг 3:      ААБВ(6)
> ```
>
> **Визуализация дерева:**
> ```
>           (6)
>          /    \
>        А(4)  (2)
>              /  \
>            Б(1) В(1)
> ```
>
> **Этап 3. Кодирование** (лево = 0, право = 1):
>
> | Символ | Код | Длина кода |
> |:---:|:---:|:---:|
> | **А** | `0` | 1 бит |
> | **Б** | `10` | 2 бита |
> | **В** | `11` | 2 бита |
>
> **Результат сжатия:**
> ```
> Было:  ААААБВ  (6 символов × 8 бит = 48 бит, при кодировке ASCII)
> Стало: 0 0 0 0 10 11  (10 бит)
> Коэффициент сжатия: 48 / 10 = 4.8×
> ```

### Пример с неравномерным текстом

> [!example] Сжатие английского текста
> Для текста `"THIS_IS_A_TEST"`:
>
> | Символ | Частота | Код Хаффмана |
> |:---:|:---:|:---:|
> | `_` | 4 | `00` |
> | `T` | 3 | `01` |
> | `S` | 2 | `100` |
> | `I` | 1 | `101` |
> | `H` | 1 | `110` |
> | `A` | 1 | `1110` |
> | `E` | 1 | `1111` |
>
> Символ `_` (самый частый) получил самый короткий код, а редкие символы — более длинные.

### Математическая основа

> [!note] Оптимальность
> Для заданного распределения частот код Хаффмана даёт ==наиболее компактный префиксный код==. Средняя длина кода:
>
> $$\bar{L} = \sum_{i=1}^{n} p_i \cdot l_i$$
>
> где $p_i$ — вероятность (частота) символа, а $l_i$ — длина его кода. Эта величина всегда находится в пределах:
>
> $$H \leq \bar{L} < H + 1$$
>
> где $H$ — энтропия Шеннона. То есть код Хаффмана не хуже энтропии более чем на 1 бит на символ.

### Адаптивный Хаффман

> [!info] Улучшение
> Существует **адаптивная версия** алгоритма Хаффмана, которая перестраивает дерево кодов «на лету» по мере считывания данных. Это позволяет:
> - Не хранить таблицу частот в архиве
> - Адаптироваться к изменениям в данных
> - Начинать сжатие без предварительного прохода по файлу

## Алгоритм Deflate

> [!abstract] Что такое Deflate?
> **Deflate** — это гибридный алгоритм сжатия, который ==объединяет LZ77 и Хаффмана== в единый мощный метод. Является стандартным алгоритмом сжатия для ZIP, gzip, PNG и многих других форматов.

### Как это работает вместе

> [!tip] Двухэтапный процесс
> **Этап 1 — LZ77**: исходные данные сжимаются алгоритмом Лемпеля-Зива, который заменяет повторяющиеся последовательности ссылками `(расстояние, длина)`. На выходе получается поток из литеральных байтов и пар `(расстояние, длина)`.
>
> **Этап 2 — Хаффман**: полученный поток кодируется алгоритмом Хаффмана. Часто встречающиеся литералы и пары получают короткие коды, а редкие — длинные. Это даёт ==дополнительное сжатие== поверх LZ77.

### Визуализация процесса

```
Исходные данные:  "АБАБАБАБАБВАБАБ"
        │
        ▼
  ┌─────────────────┐
  │    Этап 1: LZ77  │  → (0,0,А)(0,0,Б)(2,10)(0,0,В)(2,2)
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │  Этап 2: Хаффман │  → двоичный поток с переменной длиной кодов
  └────────┬────────┘
           │
           ▼
     Сжатые данные (меньший объём)
```

### Структура сжатых данных Deflate

Deflate использует два набора Хаффмановских деревьев:

1. **Дерево для литеральных байтов** (0–255) и длиной совпадений (257–285)
2. **Дерево для расстояний** (0–29)

> [!note] Деталь реализации
> Для ещё большей оптимизации Deflate может использовать **три режима** кодирования блока:
> - **Без сжатия** — данные хранятся как есть (для уже сжатых данных)
> - **Сжатие с фиксированными кодами Хаффмана** — предопределённые коды, нет overhead на хранение дерева
> - **Сжатие с динамическими кодами Хаффмана** — дерево строится под конкретные данные и сохраняется в архиве

## Другие алгоритмы сжатия

### LZ78 и LZW

> [!info] Эволюция LZ
> **LZ78** (1978) — развитие идеи LZ77. Вместо скользящего окна использует ==растущий словарь== фраз, которые встречаются в данных. Каждый раз, когда встречается новая фраза, она добавляется в словарь.
>
> **LZW** (Lempel-Ziv-Welch, 1984) — практическая реализация LZ78, разработанная Терри Велчем. Использовался в формате GIF и UNIX-утилите `compress`.

### LZMA

> [!info] Современный подход
> **LZMA** (Lempel-Ziv-Markov chain Algorithm, 2001) — алгоритм, используемый в формате **7-Zip**. Применяет LZ77 с очень большим словарём (до 4 ГБ) и **арифметическое кодирование** вместо Хаффмана для ещё более эффективного сжатия. Обеспечивает ==значительно лучшее сжатие==, чем Deflate, но требует больше памяти и процессорного времени.

### Арифметическое кодирование

> [!note] Преемник Хаффмана
> В отличие от Хаффмана, который кодирует каждый символ целым числом битов, арифметическое кодирование представляет всё сообщение как **единое дробное число** в интервале $[0, 1)$. Это позволяет приблизиться к теоретическому пределу Шеннона ==гораздо ближе==, чем кодирование Хаффмана.

## Где применяется сжатие?

> [!tip] Сжатие повсюду
> Алгоритмы сжатия данных окружают нас повсюду — мы используем их каждый день, часто даже не замечая этого.

### Веб и интернет

| Технология | Алгоритм | Зачем |
|:---|:---|:---|
| **HTTP/HTTPS** | gzip, Brotli, Zstandard | Сжатие ответов сервера — экономия трафика |
| **JPEG** | Хаффман + ДКП | Сжатие изображений (с потерями) |
| **PNG** | Deflate | Сжатие изображений без потерь |
| **MP3 / AAC** | Хаффман + MDCT | Сжатие аудио (с потерями) |
| **WebSocket** | permessage-deflate | Сжатие сообщений в реальном времени |
| **TLS 1.3** | Разные | Обязательная поддержка сжатия |

### Хранение данных

- [x] **ZIP** — архивация файлов для передачи
- [x] **7-Zip** — максимальное сжатие для хранения
- [x] **tar.gz** — резервное копирование в Linux
- [x] **APK** (Android) — по сути ZIP-архив
- [x] **JAR / DOCX / XLSX / PPTX** — тоже ZIP-архивы!

> [!example] Вы этого не знали?
> Файлы `.docx`, `.xlsx`, `.pptx`, `.odt`, `.epub`, `.apk` — это ==обычные ZIP-архивы==. Вы можете переименовать любой из этих файлов в `.zip` и открыть его в любом архиваторе, чтобы увидеть содержимое!

### Операционные системы

- [x] **NTFS** (Windows) — сжатие файлов на уровне файловой системы
- [x] **APFS** (macOS) — прозрачное сжатие
- [x] **ZFS** (Solaris/Linux) — встроенное сжатие LZ4 / Zstandard
- [x] **Btrfs** (Linux) — поддержка LZO и Zstandard

---

> [!quote] Итог
> Объединение двух методов — **поиска повторяющихся последовательностей** (алгоритм Лемпеля-Зива) и **эффективного кодирования символов** (алгоритм Хаффмана) — стало основой классического алгоритма сжатия ZIP под названием **Deflate**. Эта комбинация, зародившаяся в 1970–80-х годах, до сих пор является фундаментом большинства технологий сжатия, которые мы используем каждый день.
