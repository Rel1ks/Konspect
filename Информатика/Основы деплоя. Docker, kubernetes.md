## Введение

> [!abstract] Определение
> **Docker** и **Kubernetes** — это технологии, которые часто используются в современном деплое приложений. Docker отвечает за упаковку приложения в контейнер, а Kubernetes — за управление большим количеством таких контейнеров.

Если обычный диплой отвечает на вопрос, как разместить приложение в рабочей среде, то Docker и Kubernetes помогают сделать этот процесс ==повторяемым, удобным, масштабируемым и более предсказуемым==.

> [!quote] Ключевая идея
> Docker упаковывает приложение, а Kubernetes организует его запуск, обновление и масштабирование.

---

## Что такое Docker

> [!abstract] Определение
> **Docker** — это платформа для создания, запуска и доставки приложений в контейнерах.

Контейнер — это изолированная среда, в которой приложение запускается вместе со всем необходимым:

- кодом;
- библиотеками;
- зависимостями;
- системными пакетами;
- конфигурацией.

### Зачем нужен Docker

> [!info] Практический смысл
> Docker помогает избежать ситуации, когда приложение работает у разработчика локально, но не работает на сервере.

Преимущества Docker:

| Преимущество | Что это дает |
|:---|:---|
| **Одинаковое окружение** | Приложение запускается одинаково локально и на сервере |
| **Изоляция** | Контейнеры не мешают друг другу |
| **Переносимость** | Контейнер можно запускать на разных машинах |
| **Удобство деплоя** | Проще выкладывать готовую среду, а не настраивать все вручную |
| **Повторяемость** | Один и тот же образ можно использовать многократно |

---

## Что такое контейнер

> [!note] Базовое понятие
> **Контейнер** — это запущенный экземпляр образа, внутри которого работает приложение.

Проще говоря:

| Понятие | Значение |
|:---|:---|
| **Образ (image)** | Шаблон приложения и его окружения |
| **Контейнер (container)** | Запущенная копия образа |

```text
Dockerfile -> Image -> Container
```

### Пример логики

> [!example] Простое сравнение
> Можно представить, что:
>
> - **образ** — это подготовленный шаблон;
> - **контейнер** — это уже работающий экземпляр по этому шаблону.

---

## Что такое Dockerfile

> [!abstract] Определение
> **Dockerfile** — это файл с инструкциями, по которым Docker собирает образ приложения.

В Dockerfile обычно указывают:

- базовый образ;
- рабочую директорию;
- копирование файлов;
- установку зависимостей;
- команду запуска.

Пример логики:

```text
1. Взять базовую среду
2. Скопировать проект
3. Установить зависимости
4. Указать команду запуска
```

> [!tip] Практический смысл
> Dockerfile делает процесс сборки окружения формальным и повторяемым.

---

## Как Docker связан с деплоем

> [!info] Роль в выкладке
> В деплое Docker позволяет переносить не просто исходный код, а уже подготовленное приложение в собранной среде.

Обычный подход без Docker:

- установить зависимости на сервере;
- настроить окружение вручную;
- следить за совпадением версий;
- вручную исправлять конфликты среды.

Подход с Docker:

- собрать образ один раз;
- передать образ на сервер или в registry;
- запустить контейнер;
- получить одинаковый результат в разных средах.

---

## Что такое Kubernetes

> [!abstract] Определение
> **Kubernetes** — это система оркестрации контейнеров, которая управляет их запуском, масштабированием, обновлением и восстановлением.

Если Docker хорошо работает для упаковки и запуска одного или нескольких контейнеров, то Kubernetes нужен, когда приложение становится больше и сложнее:

- много сервисов;
- много контейнеров;
- высокая нагрузка;
- несколько серверов;
- необходимость автоматического восстановления.

> [!quote] Ключевая идея
> Kubernetes не заменяет Docker как идею контейнеризации, а берет на себя **управление контейнерами на уровне инфраструктуры**.

---

## Зачем нужен Kubernetes

> [!info] Практический смысл
> Kubernetes нужен тогда, когда контейнеров становится слишком много, чтобы управлять ими вручную.

Он помогает:

| Задача | Что делает Kubernetes |
|:---|:---|
| **Масштабирование** | Увеличивает или уменьшает количество экземпляров приложения |
| **Автовосстановление** | Перезапускает контейнеры при сбоях |
| **Балансировка** | Распределяет нагрузку между экземплярами |
| **Rolling update** | Обновляет приложение постепенно, без полного отключения |
| **Управление кластером** | Координирует работу приложений на нескольких узлах |

---

## Основные понятия Kubernetes

> [!abstract] Базовые элементы
> Для понимания Kubernetes полезно знать несколько ключевых объектов.

| Объект | Значение |
|:---|:---|
| **Cluster** | Совокупность серверов, где работает Kubernetes |
| **Node** | Отдельная машина внутри кластера |
| **Pod** | Минимальная единица запуска, обычно включает контейнер |
| **Deployment** | Описание того, как запускать и обновлять приложение |
| **Service** | Способ доступа к приложению внутри или вне кластера |
| **ConfigMap / Secret** | Хранение конфигурации и секретных данных |

```text
Cluster
  |
  +-- Node
  |    |
  |    +-- Pod
  |         |
  |         +-- Container
  |
  +-- Service
  +-- Deployment
```

---

## Что такое Pod

> [!note] Важное понятие
> **Pod** — это базовая единица размещения в Kubernetes. Обычно внутри pod находится один основной контейнер приложения, но контейнеров может быть несколько.

Почему Kubernetes работает через pod, а не напрямую через контейнер:

- pod задает сетевое окружение;
- pod объединяет связанные контейнеры;
- pod выступает единицей управления и перезапуска.

---

## Что такое Deployment

> [!abstract] Определение
> **Deployment** описывает, сколько экземпляров приложения должно работать и как их нужно обновлять.

Он позволяет:

- запускать нужное количество копий;
- автоматически заменять упавшие pod;
- обновлять приложение по новой версии;
- откатываться при неудачном обновлении.

> [!tip] Практический смысл
> Deployment делает запуск приложения в Kubernetes управляемым и устойчивым.

## Docker и Kubernetes вместе

> [!info] Как они связаны
> Эти технологии часто используются вместе, но выполняют разные роли.

| Технология | Роль |
|:---|:---|
| **Docker** | Создает и упаковывает приложение в контейнер |
| **Kubernetes** | Управляет запуском и жизненным циклом контейнеров |

### Общая цепочка

```text
Код приложения
      |
      v
Dockerfile
      |
      v
Docker image
      |
      v
Container registry
      |
      v
Kubernetes cluster
      |
      v
Запущенные pod и service
```

## Как выглядит деплой с Docker и Kubernetes

> [!example] Упрощенный сценарий
> В реальном проекте процесс может выглядеть так:

1. Разработчик пишет приложение.
2. Создается `Dockerfile`.
3. Docker собирает образ приложения.
4. Образ отправляется в registry.
5. Kubernetes получает описание deployment.
6. Кластер запускает нужное число pod.
7. Service открывает доступ к приложению.
8. При обновлении запускается новая версия без полного простоя.

## Возможные сложности

> [!warning] Что важно учитывать
> Несмотря на преимущества, Docker и Kubernetes добавляют и дополнительную сложность.

Типичные трудности:

- нужно понимать контейнеризацию;
- нужно уметь работать с образами и registry;
- Kubernetes сложнее обычного ручного деплоя;
- требуется настройка сети, конфигурации и секретов;
- ошибки в orchestration могут быть труднее для диагностики.

> [!note] Практический вывод
> Docker полезен уже на небольших проектах, а Kubernetes особенно оправдан там, где есть рост нагрузки, несколько сервисов и необходимость масштабирования.

## Где применяются Docker и Kubernetes

> [!info] Типичные случаи использования
> Эти технологии особенно распространены в современной backend- и cloud-разработке.

Они применяются в:

- web-приложениях;
- API и backend-сервисах;
- микросервисной архитектуре;
- облачной инфраструктуре;
- CI/CD-конвейерах;

## Вывод

> [!abstract] Краткий итог
> **Docker** и **Kubernetes** являются важными инструментами современного деплоя.
>
> Docker позволяет упаковать приложение в контейнер вместе с окружением, а Kubernetes управляет запуском, масштабированием, обновлением и восстановлением таких контейнеров.
>
> Вместе они помогают сделать деплой более стабильным, переносимым, автоматизированным и удобным для развития сложных систем.
